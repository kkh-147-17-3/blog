# Next.js + Vercel 개인 블로그 기획문서

## 개요
본 문서는 Next.js + Vercel 기반의 개인 블로그 구축 방향을 정리한 기획문서다. 대상 블로그는 기술 글뿐 아니라 일기, 회고, 일상 기록까지 함께 다루는 다목적 개인 블로그이며, 디자인 자유도와 장기적인 확장성을 우선한다.[cite:102][cite:107]

이번 방향에서는 Notion 연동은 제외하고, Next.js 내부에 관리자 페이지를 직접 두는 구조를 채택한다. 이 방식은 브라우저 기반 글쓰기 경험을 제공하면서도, 프론트엔드와 데이터 모델을 한 저장소 안에서 일관되게 관리할 수 있다.[cite:107][cite:110]

## 목표
이 프로젝트의 목표는 다음과 같다.

- Next.js + Vercel 조합으로 디자인 커스터마이징이 자유로운 개인 블로그 구축.[cite:102]
- 기술 글, 일기, 회고, 메모 등 여러 성격의 글을 한 플랫폼 안에서 관리.[cite:107]
- 코드 에디터 없이도 브라우저에서 글 작성, 수정, 발행 가능.
- 조회수, 댓글, 좋아요 같은 동적 기능을 점진적으로 추가 가능한 구조 확보.[cite:107][cite:115]
- 무료 플랜 범위 안에서 시작하고, 필요 시 유료 전환이 자연스러운 구조 채택.[cite:102][cite:115]

## 왜 이 구조인가
GitHub 웹 에디터는 별도 관리자 개발이 필요 없다는 장점이 있지만, 일반 글쓰기 경험이 좋지 않다. Markdown 문법을 직접 다뤄야 하고, 이미지 업로드와 글감 정리, 임시저장, 공개/비공개 제어 같은 블로그 운영 기능이 빈약하다.

반면 관리자 페이지를 직접 만드는 방식은 초기 개발 공수가 더 크지만, 장기적으로는 글쓰기 경험과 기능 확장성에서 훨씬 유리하다. 특히 일기나 사적인 기록처럼 자주 쓰고 빠르게 저장해야 하는 콘텐츠는 웹 기반 에디터가 더 적합하다.

## 시스템 구성
전체 구조는 다음과 같다.

```text
사용자 브라우저
  ├─ 공개 블로그 페이지
  └─ 관리자 페이지 (/admin)
          ↓
Next.js App Router
  ├─ 페이지 렌더링
  ├─ Route Handlers (/app/api/*)
  └─ 인증/권한 체크
          ↓
Supabase
  ├─ PostgreSQL (글, 댓글, 조회수, 태그 등)
  ├─ Storage (이미지 업로드)
  └─ Auth (관리자 로그인)
          ↓
Vercel
  ├─ 프론트엔드 배포
  └─ 서버리스 함수 실행
```

Next.js App Router는 페이지와 API를 같은 프로젝트 안에서 관리할 수 있고, Route Handlers를 통해 GET, POST, PUT, DELETE 같은 HTTP 메서드 기반 API를 직접 구성할 수 있다.[cite:107][cite:110]

## 핵심 기술 스택
| 영역 | 선택안 | 비고 |
|---|---|---|
| 프론트엔드 | Next.js App Router | 페이지와 API 통합 가능[cite:107] |
| 배포 | Vercel Hobby | 개인 프로젝트 무료 시작 가능, 월 100GB 대역폭[cite:102][cite:104] |
| DB | Supabase PostgreSQL | 게시글, 댓글, 조회수 저장[cite:115][cite:109] |
| 파일 저장 | Supabase Storage | 본문 이미지, 썸네일 관리[cite:115] |
| 인증 | Supabase Auth | 관리자 로그인용[cite:115] |
| 관리자 에디터 | 자체 WYSIWYG 에디터 | 블로그 목적에 맞춰 직접 설계 |
| 통계/동적 기능 | Next.js Route Handlers + Supabase | 조회수, 좋아요, 댓글 등[cite:107][cite:115] |

## 관리자 방식 결정
### WYSIWYG 관리자 페이지 채택
권장안은 `/admin` 페이지를 직접 구현하는 것이다. 관리자는 로그인 후 브라우저에서 글을 작성하고, 제목, 카테고리, 공개 여부, 태그, 썸네일, 본문을 입력한 뒤 저장 또는 발행할 수 있다.

이 방식은 다음 조건에 특히 적합하다.

- 기술 글과 일상 글을 함께 다룬다.
- 모바일 또는 외부 PC에서도 글을 쓰고 싶다.
- 공개/비공개, 임시저장, 카테고리 분리 기능이 필요하다.
- 장기적으로 댓글, 좋아요, 조회수, 방명록을 같은 관리 화면에서 보고 싶다.

### GitHub 웹 에디터와 비교
| 항목 | 자체 관리자 페이지 | GitHub 웹 에디터 |
|---|---|---|
| 글쓰기 편의성 | 높음 | 낮음 |
| 이미지 업로드 | 쉬움 | 불편 |
| 비공개/임시저장 | 구현 가능 | 사실상 불편 |
| 일기/짧은 기록 작성 | 적합 | 부적합 |
| 버전 관리 | 별도 설계 필요 | Git 기반으로 강함 |
| 초기 개발 비용 | 높음 | 낮음 |
| 장기 확장성 | 매우 높음 | 낮음 |

프로젝트 목표가 다목적 개인 블로그라면, GitHub 웹 에디터보다 자체 관리자 페이지가 더 적합하다.

## 주요 기능
### 1. 공개 블로그 기능
- 홈: 최신 글 피드, 카테고리별 섹션, 추천 글.
- 글 목록: 기술, 일기, 회고, 메모 등 카테고리별 아카이브.
- 글 상세: 본문, 태그, 작성일, 읽기 시간, 조회수, 관련 글.
- 검색: 제목, 본문, 태그 기반 검색.
- 다크모드: 읽기 경험 향상.
- RSS, sitemap, SEO 메타데이터 자동 생성.

### 2. 관리자 기능
- 로그인/로그아웃.
- 글 작성, 수정, 삭제.
- 임시저장, 발행, 비공개 설정.
- 카테고리, 태그, 시리즈 관리.
- 대표 이미지 업로드.
- 댓글 관리 및 숨김 처리.
- 조회수/반응 통계 확인.

### 3. 동적 기능
조회수, 댓글, 좋아요 같은 기능은 Next.js Route Handlers와 Supabase를 조합해 구현한다. Next.js는 앱 내부에서 API 엔드포인트를 만들 수 있으므로 별도 백엔드 서버 없이도 동적 기능을 추가할 수 있다.[cite:107][cite:110]

예시 기능은 다음과 같다.

- 조회수: 글 진입 시 slug 기준 카운트 증가.
- 좋아요: 사용자 또는 세션 기준 중복 제한.
- 댓글: 익명 또는 닉네임 기반 댓글 작성.
- 방명록: 별도 페이지에서 운영.
- 뉴스레터 구독: 이메일 저장 후 추후 발송 기능 확장.

## 인증 및 권한
관리자 인증은 Supabase Auth로 처리한다. 일반 방문자는 읽기 전용 권한만 갖고, `/admin` 이하 경로 및 쓰기 API는 관리자 계정만 접근 가능하게 제한한다.[cite:115]

추가로 Row Level Security를 설정해 클라이언트에서 직접 DB를 호출하더라도 권한 없는 수정이 불가능하도록 막는 것이 바람직하다.[cite:115]

## 에디터 설계 방향
에디터는 단순 textarea보다 블록 기반 WYSIWYG 구조가 적합하다. 제목, 본문, 이미지, 코드 블록, 인용문, 체크리스트, 구분선 정도를 기본 블록으로 제공하면 기술 글과 일상 글을 모두 커버하기 쉽다.

권장 요구사항은 다음과 같다.

- 제목, 소제목, 본문, 리스트, 체크리스트.
- 코드 블록 언어 선택.
- 이미지 업로드 및 본문 삽입.
- 자동 저장.
- 미리보기.
- 공개/비공개/임시저장 전환.
- URL slug 자동 생성 및 수동 수정.

초기 구현은 Tiptap 같은 에디터 라이브러리를 검토할 수 있지만, 제품 관점에서는 "블록 기반 관리자 에디터"라는 요구만 먼저 확정해도 충분하다.

## 배포 및 운영
Vercel Hobby 플랜은 개인 프로젝트 기준 무료로 시작할 수 있고, 월 100GB 대역폭이 제공된다.[cite:102][cite:104] 개인 블로그 초기 운영에는 충분한 수준이며, Next.js와의 통합도 가장 자연스럽다.[cite:102]

Supabase Free 플랜은 500MB 데이터베이스, 1GB 파일 저장소, 5GB egress, 50,000 월간 활성 사용자 등의 범위를 제공한다.[cite:109][cite:115] 텍스트 중심 개인 블로그에는 충분하지만, 이미지 업로드가 많아지면 Storage 사용량은 별도로 모니터링해야 한다.[cite:109]

## 무료 플랜 기준 운영 판단
| 항목 | 무료 범위 | 판단 |
|---|---|---|
| Vercel 대역폭 | 100GB/월[cite:102][cite:104] | 개인 블로그에 대체로 충분 |
| Vercel 배포 | Git 기반 자동 배포[cite:102] | 충분 |
| Supabase DB | 500MB[cite:109] | 텍스트 중심이면 충분 |
| Supabase 파일 저장소 | 1GB[cite:109] | 이미지 최적화 필요 |
| Supabase MAU | 50,000[cite:109] | 관리자/방문자 규모에 충분 |

## 리스크와 대응
### 1. 초기 개발 범위가 커질 수 있음
관리자 페이지를 직접 만들면 글쓰기 UX까지 직접 설계해야 하므로 생각보다 범위가 커질 수 있다. 이를 막기 위해 초기에는 "글 작성/수정/발행"에 집중한 MVP부터 완성하는 것이 좋다.

### 2. WYSIWYG 데이터 구조 결정
HTML 저장, Markdown 저장, JSON 블록 저장 중 무엇을 원본으로 둘지 초기에 정해야 한다. 장기 확장성은 JSON 원본 + HTML 캐시 조합이 유리하다.

### 3. 이미지 관리 비용
이미지 업로드가 많아지면 Supabase Storage 무료 한도 소진이 빨라질 수 있다.[cite:109] 업로드 시 리사이징, WebP 변환, 썸네일 자동 생성 전략을 초기에 고려하는 것이 좋다.

## 최종 제안
이 프로젝트의 권장 방향은 **Next.js + Vercel + Supabase + 자체 관리자 페이지** 조합이다. 이 구성은 디자인 자유도, 글쓰기 편의, 동적 기능 확장성의 균형이 가장 좋다.[cite:102][cite:107][cite:115]

특히 GitHub 웹 에디터는 기술 블로그 하나만 운영할 때는 간단한 대안이 될 수 있지만, 기술 글과 일기를 함께 다루는 다목적 개인 블로그에는 맞지 않는다. 다양한 목적의 글을 장기적으로 운영할 계획이라면 자체 관리자 페이지가 더 적합하다.

## 권장 의사결정
- 배포: Vercel.
- 프레임워크: Next.js App Router.[cite:107]
- 데이터 저장: Supabase PostgreSQL + Storage.[cite:115]
- 관리자 작성 도구: 자체 WYSIWYG 에디터.
- 초기 범위: 글 CRUD + 이미지 업로드 + 공개/비공개 + 조회수.
- 확장 순서: 댓글 → 좋아요 → 검색 → 뉴스레터.
